# mse-template
## О проекте
**Сервис для автоматического подбора гиперпараметров моделей YOLO** с удобным веб-интерфейсом, запуском экспериментов и сохранением всех результатов (веса, графики, метрики).
### Постановка задачи
Разработать интеллектуальную веб-платформу AutoML для компьютерного зрения, которая принимает датасет, запускает серию экспериментов (HPO Model Search), сравнивает результаты и выдает лучшую YOLO-модель с полной воспроизводимостью.
Проект решает проблему ручного перебора параметров обучения и архитектур (n/s/m/l), который требует большого опыта, множества запусков.
### Требования
Функциональные:

    Загрузка пути к локальному датасету и автоматическая валидация его структуры.

    Определение пространства поиска (Model Search, HPO, Augmentations).

    Поддержка алгоритмов Grid Search и Random Search.

    Формирование очереди обучения на GPU и сравнение метрик.

    Экспорт лучшей модели (веса .pt, конфигурация, артефакты).

Нефункциональные:

    Ограничения: последовательная обработка очереди на одной видеокарте, лимиты времени вычислений.

    Метрики успеха: mAP, Precision, Recall, FPS, размер модели.

    Стек: Python, FastAPI, Ultralytics YOLO, CUDA, JS/HTML/CSS для SPA.
### Сценарии использования
Сценарий 1: Базовый сценарий (Успешный подбор модели)
================================================================================
Описание: Основной путь , при котором система находит оптимальную модель без сбоев.

1.  Инициализация: Пользователь открывает веб-страницу локально в браузере.  Регистрация для работы не требуется.
2. Настройка параметров: Пользователь вводит путь к датасету.  Датасет не требует дополнительной обработки, так как уже разделен на выборки train/test/val.
3.  Выбор алгоритма: Пользователь выбирает метод перебора гиперпараметров, достаточно базовых Grid Search или Random Search.
4.  Запуск: Фронтенд отправляет данные, и система начинает последовательное обучение моделей, подразумевая запуск на одной видеокарте.
5.  Обучение и Мониторинг: Перебор запускается в фоне.  В интерфейсе отображаются логи, при этом обработка видео не производится, нужны только логи.
6. Завершение: После перебора всех комбинаций система выдает результат.  Пользователь скачивает только файл лучшей модели в формате .pt.

Сценарий 2: Срабатывание ранней остановки 
================================================================================
Описание: Защита системы от пустой траты вычислительных ресурсов на неудачные конфигурации.

1.  Старт эпохи: Запускается обучение очередной модели из очереди.
2.  Отслеживание метрик: Парсер читает файл results.csv и собирает значения train/box_loss в конце каждой эпохи.
3.  Прохождение порога: Модель обучается минимум 20 эпох.
4.  Фиксация деградации: Алгоритм фиксирует, что значение Loss резко пошло вверх   — выросло на 15% от минимума за последние 3 эпохи.
5.  Прерывание: Срабатывает коллбэк on_fit_epoch_end, выполняется команда trainer.stop = True. Обучение текущей модели останавливается, система переходит к следующей конфигурации.

Сценарий 3: Блокировка из-за ошибки валидации данных
================================================================================
Описание: Предотвращение падения ML-ядра из-за некорректных входных данных до старта обучения.

1.  Запуск с битыми данными: Пользователь указывает путь к датасету и нажимает кнопку старта.
2.  Проверка изображений: Скрипт рекурсивно проверяет входящие изображения на физическую целостность.  Учитывая требования, видеофайлы мы игнорируем и не обрабатываем.
3.  Проверка разметки: Запускается обработчик для текстовых файлов labels.  Он проверяет формат для задачи обнаружения (детекции).
4.  Обнаружение ошибки: Скрипт находит строку, где отсутствуют правильные данные: идентификатор класса, нормализованные координаты центра, ширина или высота bounding box.
5.  Остановка: Процесс обучения блокируется до запуска YOLO. Выводится ошибка, предотвращающая падение системы в процессе обучения.

## Способы запуска

### 1. Docker + GPU (рекомендуется для всех)

```bash
# Сборка и запуск в фоне
docker compose up --build -d

# Перезапуск после изменений кода
docker compose restart

# Полная остановка
docker compose down
```

### 2. Локальный запуск (только для разработки)

```bash
cd markup_ml

# Виртуальное окружение
python -m venv .venv
# Windows:
.venv\Scripts\activate
# Linux/macOS:
source .venv/bin/activate

pip install -r requirements.txt

uvicorn main:app --reload --host 0.0.0.0 --port 8000
```

## Проверка после запуска

Откройте в браузере:

- **Главная страница фронтенда**: [http://localhost:8000](http://localhost:8000)
- **Swagger-документация**: [http://localhost:8000/docs](http://localhost:8000/docs)
- **Пинг**:  

  ```bash
  curl http://localhost:8000/ping
  ```

  Ответ: `{"status": "ok"}`

**Результаты экспериментов** доступны по URL:

- `http://localhost:8000/runs/detect/exp1/best.pt`
- `http://localhost:8000/runs/detect/exp1/results.png`

---

## Работа с результатами (`runs/`)

- Папка `runs/` монтируется как volume → все данные сохраняются на хосте
- Доступ только на чтение через HTTP
- Структура результатов соответствует Ultralytics YOLO (папки `detect/`, `train/` и т.д.)
- Если папки нет — создайте её вручную: `mkdir -p runs`

---

## Сборка / обновление фронтенда

1. Соберите фронтенд в папку `dist` или `build` (в отдельном репозитории)
2. Скопируйте всё содержимое в `./static/`
3. Перезапустите сервис:

   ```bash
   docker compose restart
   ```
